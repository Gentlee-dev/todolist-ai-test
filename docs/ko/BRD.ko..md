## 1. 프로젝트 개요

### 1.1 프로젝트 목적
쇼핑몰 본 프로젝트에 앞서 **AI 자동화 파이프라인(기획 → 디자인 → 개발)**의 실효성을 검증하기 위한 파일럿 프로젝트. TodoList 앱을 통해 다음을 테스트한다:

- 기획 문서(BRD → PRD) 기반 AI 개발 워크플로우
- v0 프로토타이핑 → Claude Code 개발 파이프라인
- 디자인 토큰 시스템의 일관성 유지 능력
- shadcn/ui + Next.js App Router 기술 스택 검증

### 1.2 프로젝트 범위
- **In Scope**:
  - 프론트엔드 TodoList 앱 (Next.js)
  - 백엔드 REST API (Node.js)
  - 모노레포 구조 (프론트 + 백 단일 저장소)
  - Docker 컨테이너화 (개발 환경 + DB)
  - 로컬 Docker Compose로 전체 스택 실행
- **Out of Scope**: 인증/인가, 클라우드 배포, CI/CD

### 1.3 성공 기준
| 기준 | 측정 방법 |
|------|-----------|
| 파이프라인 전 과정 완주 | BRD → PRD → 디자인 → 개발 완료 여부 |
| 디자인 토큰 재활용 가능성 | 쇼핑몰 프로젝트에 토큰을 그대로 적용 가능한지 |
| Claude Code 자동화 효율 | claude.md 기반으로 일관된 코드를 생성하는지 |
| 모노레포 구조 검증 | 프론트/백 공유 타입이 원활히 동작하는지 |
| Docker 개발 환경 | `docker compose up` 한 번으로 전체 스택 기동 가능한지 |
| 병목 구간 식별 | 파이프라인에서 사람 개입이 필요한 지점 파악 |

## 2. 비즈니스 요구사항

### 2.1 핵심 기능 요구사항

#### F1. 할 일 CRUD
- **생성**: 할 일 텍스트 입력 후 추가
- **조회**: 전체 할 일 목록 표시, 완료/미완료 구분
- **수정**: 할 일 텍스트 인라인 편집
- **삭제**: 개별 삭제 + 완료 항목 일괄 삭제

#### F2. 카테고리/태그 시스템
- 카테고리 생성, 수정, 삭제
- 할 일에 카테고리 1개 지정 (선택사항)
- 태그 생성 (자유 입력)
- 할 일에 태그 N개 지정 (선택사항)
- 카테고리별, 태그별 필터링

#### F3. 드래그앤드롭 정렬
- 할 일 항목을 드래그하여 순서 변경
- 변경된 순서 서버에 저장 (Optimistic Update 적용)
- 모바일 터치 드래그 지원

### 2.2 비기능 요구사항

| 항목 | 요구사항 |
|------|----------|
| 데이터 영속성 | PostgreSQL (Docker 컨테이너) |
| 반응형 | 모바일(360px) ~ 데스크톱(1440px) |
| 접근성 | 키보드 네비게이션 기본 지원 |
| 성능 | 100개 항목에서 드래그 렉 없음 |
| 개발 환경 | Docker Compose 한 명령으로 전체 스택 기동 |

### 2.3 디자인 방향

- **쇼핑몰 톤의 디자인 토큰** 사용 (본 프로젝트 재활용 목적)
- 깔끔하고 모던한 커머스 UI 느낌
- shadcn/ui 컴포넌트 기반
- 다크모드 미지원 (1차 범위 외)

## 3. 기술 제약사항

### 3.1 기술 스택 (확정)

#### 프론트엔드
- **프레임워크**: Next.js 16 App Router
- **언어**: TypeScript
- **UI 라이브러리**: shadcn/ui + Tailwind CSS
- **드래그앤드롭**: @dnd-kit (React 생태계 표준)
- **상태관리**: React useState/useReducer (간단한 범위)
- **API 통신**: TanStack Query (서버 상태 관리 + Optimistic Update)

#### 백엔드
- **런타임**: Node.js 20+
- **프레임워크**: Express (Medusa.js와 동일 기반, 쇼핑몰 실전 대비)
- **언어**: TypeScript
- **ORM**: Prisma
- **DB**: PostgreSQL 16

#### 모노레포
- **도구**: Turborepo (Vercel 생태계, Next.js와 궁합 좋음)
- **구조**: `apps/web` (프론트) + `apps/server` (백엔드) + `packages/shared` (공유 타입)

#### 인프라 (로컬)
- **Docker Compose**: 전체 스택 오케스트레이션
  - `web` 컨테이너 (Next.js dev server)
  - `server` 컨테이너 (Node.js API server)
  - `db` 컨테이너 (PostgreSQL)
- **핫 리로드**: 볼륨 마운트로 코드 변경 즉시 반영

### 3.2 개발 환경
- Docker Desktop
- Node.js 20+ (로컬 개발 시)
- pnpm 패키지 매니저 (모노레포 workspace)

## 4. 제약조건 및 가정

### 제약조건
- 1인 개발 (AI 자동화 파이프라인 활용)
- 로컬 Docker 환경에서만 실행 (클라우드 배포 없음)
- 인증/인가 없음 (단일 사용자)

### 가정
- 단일 사용자 (userId 컬럼은 두되 하드코딩, 인증 없음)
- Docker Desktop이 설치되어 있음
- 브라우저: 최신 Chrome 기준
- DB 데이터는 Docker 볼륨으로 유지 (컨테이너 재시작 시에도 보존)
